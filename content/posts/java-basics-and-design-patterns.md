# Java 基础与设计模式：后端开发的必修课

Java 作为企业级后端开发的主流语言，基础扎实程度直接决定了代码质量和系统稳定性。这篇文章把 Java 语言核心、集合原理、并发基础，以及最常用的 5 个设计模式串在一起讲，目标是建立一个完整的后端知识骨架。

## Java 语言核心

### JDK、JRE、JVM 的关系

- **JVM**：Java 虚拟机，负责执行字节码
- **JRE**：运行环境 = JVM + Java 核心类库
- **JDK**：开发工具包 = JRE + 编译器 + 调试工具

一句话记忆：写代码用 JDK，跑程序靠 JRE，真正执行字节码的是 JVM。

Java 的跨平台能力来自"同一份字节码可以在不同平台的 JVM 上运行"。`.java` 源文件经 `javac` 编译成 `.class` 字节码，再由各平台 JVM 翻译成机器指令执行。

### 面向对象四大特性

| 特性 | 核心作用 | 注意点 |
|------|---------|--------|
| 封装 | 把属性和方法封装在类里，隐藏实现细节 | 用访问修饰符控制暴露范围 |
| 继承 | 子类复用父类能力 | 继承层级不要太深 |
| 多态 | 同一个父类引用，运行时表现出不同子类行为 | 编译看左边，运行看右边 |
| 抽象 | 把共性提炼出来，细节交给具体实现类 | 接口和抽象类配合使用 |

### 高频对比速记

**重载 vs 重写**

- 重载：同一个类中，同名不同参
- 重写：父子类之间，子类重新实现父类方法

**`==` vs `equals()`**

- `==`：基本类型比值，引用类型比地址
- `equals()`：默认比地址，很多类重写为比内容

重写 `equals()` 通常也要重写 `hashCode()`，否则会破坏哈希容器行为。

**接口 vs 抽象类**

- 接口更偏规范和能力扩展，一个类可实现多个接口
- 抽象类更偏共性复用和代码模板，一个类只能继承一个抽象类

### String 家族

- `String`：不可变，适合普通字符串使用
- `StringBuilder`：可变，线程不安全，单线程拼接性能高
- `StringBuffer`：可变，线程安全，多线程下可用但通常不常用

### 异常体系

```
Throwable
  ├─ Error（系统级错误，通常不处理）
  └─ Exception
       ├─ 受检异常（必须显式处理或声明）
       └─ RuntimeException（运行时异常，可选处理）
```

`try-catch-finally` 中 `finally` 通常会执行，即使 `return` 也会先走 `finally`。

## 集合框架

### List、Set、Map 的核心区别

- `List`：有序、可重复
- `Set`：通常不允许重复
- `Map`：键值对结构

### ArrayList vs LinkedList

- `ArrayList`：底层数组，随机访问快，大多数业务场景优先选择
- `LinkedList`：底层链表，插入删除某些场景更灵活

### HashMap 核心原理

底层结构：数组 + 链表，Java 8 后冲突严重时可转红黑树。

存储流程：
1. 计算 key 的 hash
2. 定位数组下标
3. 若冲突，进入链表或树继续比较
4. 若 key 已存在则覆盖，否则插入

`HashMap` 线程不安全的原因：并发写会数据覆盖，扩容时可能出问题。

### ConcurrentHashMap

面向并发场景，通过更细粒度的并发控制和 CAS 机制提升并发下的安全性与性能。

## JVM 基础

### 运行时内存区域

- **堆**：放对象实例
- **栈**：方法调用、局部变量
- **方法区 / 元空间**：类元信息、常量等

### 类加载过程

加载 -> 验证 -> 准备 -> 解析 -> 初始化

### 双亲委派模型

类加载请求先交给父加载器处理，父加载器处理不了再由子加载器处理。好处是避免重复加载，保护核心类库不被随意替换。

### GC 基础

主要回收堆中的无用对象，常见分代思想：新生代、老年代。常见收集器包括 Serial、ParNew、CMS、G1。

**内存泄漏 vs 内存溢出**

- 内存泄漏：对象已经没用了，但还被引用着，回收不掉
- 内存溢出：申请不到足够内存，直接 OOM

## 并发编程基础

### 进程和线程

- 进程：资源分配基本单位
- 线程：CPU 调度基本单位

### synchronized vs volatile

- `volatile`：保证可见性，禁止部分指令重排，不保证复合操作原子性，适合状态标记
- `synchronized`：保证互斥与可见性，适合临界区保护

### CAS

Compare And Swap，核心思想是比较当前值是否仍是预期值，如果是就更新，否则失败重试。常见于原子类和并发容器。

### 线程池

为什么要用线程池：复用线程、控制资源、提高响应速度。核心参数包括核心线程数、最大线程数、队列容量、拒绝策略。

### 死锁

死锁四条件：互斥、请求与保持、不可剥夺、循环等待。避免思路：固定加锁顺序、减少锁持有时间、尝试锁超时。

## 最常用的 5 个设计模式

设计模式是前人总结出来的"常见问题的经典解法"。对后端开发来说，先掌握最常用的 5 个模式，比泛览 23 个模式更有价值。

### 1. 单例模式

保证一个类在系统里只有一个实例，并提供统一访问入口。

常见写法：饿汉式、懒汉式、双重检查锁。面试重点要说明 `volatile` + 双重 `if` + `synchronized` 的组合作用。

适用场景：配置中心、连接池管理器、缓存管理器。

> 注意：Spring 里的单例 Bean 和传统手写 Singleton 不完全是一回事。传统单例是类自己控制实例创建，Spring 单例是容器控制"一个容器里通常只有一份 Bean"。

### 2. 工厂模式

把对象创建逻辑封装起来，让业务代码只依赖抽象，不直接依赖具体实现。

核心好处：业务和创建解耦，新增类型时不需要修改原有业务代码。

适用场景：支付处理器创建、消息发送器创建、导出器创建。

### 3. 策略模式

把一组可替换的业务规则或算法封装成不同策略类，对外暴露统一接口，运行时按条件选择具体策略。

核心价值：消除大量 `if-else`，让每种规则独立维护。

适用场景：支付方式选择、折扣规则计算、订单状态处理、渠道分发。

在 Spring 项目里，常见写法是让不同策略都实现同一接口并标记 `@Component`，再用一个上下文类统一收口。

### 4. 模板方法模式

在父类中定义算法或业务流程的整体骨架，把部分可变步骤交给子类实现。

核心特点：流程由父类定，细节由子类补。

适用场景：导出流程、审批流程、支付流程、数据同步流程。

### 5. 代理模式

通过代理对象间接访问目标对象，并在调用前后织入公共逻辑。

核心价值：不改原业务代码就能增强功能。

静态代理需要手写代理类，动态代理在运行时生成代理对象。Spring AOP、事务管理、日志增强本质上都和代理模式密切相关。

> 代理模式在面试里既是设计模式题，也是 Spring AOP 的底层理解题。

## 5 个模式速记表

| 模式 | 核心目的 | 速记口诀 | 典型场景 |
|------|---------|---------|---------|
| 单例 | 保证唯一实例 | 只要一个对象 | 配置中心、管理器 |
| 工厂 | 封装对象创建 | 对象别到处 new | 支付处理器、消息发送器 |
| 策略 | 封装不同算法 | 干掉 if-else | 折扣、订单处理、渠道分发 |
| 模板方法 | 固定流程骨架 | 父类定流程，子类补细节 | 导出、审批、同步流程 |
| 代理 | 不改原类做增强 | 前后包一层 | 日志、事务、权限、AOP |

## 复习建议

如果时间有限，按这个优先级：

1. 先把这 5 个设计模式讲顺，每个都能说出"解决什么问题"
2. 掌握 `HashMap`、`ConcurrentHashMap` 的核心原理和区别
3. 理解 `synchronized`、`volatile`、CAS 的适用场景
4. 能说清 JVM 内存区域、类加载过程和 GC 分代思想
5. 把 BIO / NIO / AIO 的同步阻塞关系讲清楚

Java 基础面试的核心不是死记 API，而是把语言特性、集合原理、JVM、并发、IO 这些基础能力串起来，形成一个完整的后端知识骨架。
