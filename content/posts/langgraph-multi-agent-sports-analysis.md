## 引言

单个大语言模型调用足以完成很多任务，但复杂场景下，单一 Agent 的能力边界很快显现。当分析任务涉及多维度数据采集、多角度观点碰撞、结果质量审查时，多 Agent 协作工作流成为更优解。

本文基于 NBA 赛前分析 Agent Demo 项目，介绍如何用 LangGraph 编排一个多节点、多角色的智能分析系统。

## 一、为什么需要多 Agent 工作流

### 单一 Agent 的局限

让一个大模型同时完成"数据整理 + 乐观分析 + 悲观分析 + 报告撰写 + 质量审查"，会出现几个问题：

- **角色混淆**：模型在乐观和悲观立场之间摇摆，输出中庸但缺乏洞察
- **上下文爆炸**：单次 Prompt 过长，关键信息被稀释
- **质量不可控**：没有独立的审查环节，错误可能直接输出

### 多 Agent 的核心优势

| 维度 | 单 Agent | 多 Agent 工作流 |
|------|---------|---------------|
| 角色专注度 | 一个模型承担所有角色 | 每个节点只负责一件事 |
| 观点多样性 | 单一视角 | 多角色辩论，输出更全面 |
| 质量保障 | 无独立审查 | 专门节点负责质检 |
| 可调试性 | 黑盒一次调用 | 可逐节点检查中间结果 |
| 可扩展性 | 修改 Prompt 影响全部 | 新增节点不影响现有流程 |

## 二、LangGraph 核心概念

### 状态机模型

LangGraph 把 Agent 工作流建模为**状态机**：

- **State**：整个工作流的共享状态对象，各节点可以读取和修改
- **Node**：工作流中的一个步骤，接收当前 State，返回更新后的 State
- **Edge**：节点之间的连接，分为普通边和条件边
- **Graph**：节点和边的集合，定义完整的执行流程

### 与普通链式调用的区别

LangChain 的 `RunnableSequence` 是线性管道，A -> B -> C，适合简单场景。LangGraph 支持：

- **分支**：根据条件走不同路径
- **循环**：支持回退重试（如审查不通过返回修改）
- **并行**：多个节点同时执行，结果汇聚

## 三、NBA 赛前分析工作流设计

### 整体流程

```text
START
  -> load_fixture          (加载比赛数据)
  -> analyze_team_form      (分析球队状态)
  -> analyze_injuries       (分析伤病影响)
  -> analyze_matchup        (分析对位关系)
  -> [并行]
       -> bullish_analyst   (乐观分析师)
       -> cautious_analyst  (谨慎分析师)
  -> synthesize_report     (综合报告)
  -> review_report         (质量审查)
       -> 通过 -> finalize  (输出终稿)
       -> 不通过 -> synthesize_report (最多重试 1 次)
  -> END
```

### 节点职责详解

#### load_fixture：数据加载

从 ESPN API 或本地 mock 数据加载比赛信息，包括：

- 比赛时间、场馆、主客队
- 球队战绩、近五场结果
- 基础统计数据
- 伤病名单
- 历史交锋摘要

输出：结构化的 `fixture` 对象，供后续节点使用。

#### analyze_team_form：状态分析

分析主客队近期的竞技状态：

- 近五场胜负走势
- 主客场表现差异
- 攻防效率变化趋势
- 赛程密度影响

**设计原则**：这个节点只做"状态整理"，不做"预测判断"。预测留给专门的分析师节点。

#### analyze_injuries：伤病评估

评估伤病名单对比赛的影响：

- 核心球员缺阵的战术影响
- 替补球员的填补能力
- 伤病对团队士气的影响

#### analyze_matchup：对位分析

分析主客队的战术对位关系：

- 核心球员对位优劣势
- 进攻体系 vs 防守体系的克制关系
- 教练战术调整历史

#### bullish_analyst：乐观分析师

**角色设定**：看好主队的分析师，职责是从数据中找出支持主队获胜的论据。

```
你是一位看好 {主队} 的分析师。
你的任务是：基于提供的数据，找出所有支持 {主队} 获胜的理由。
要求：
1. 每个论点必须有数据支撑
2. 承认对你不利的数据，但给出反驳角度
3. 输出 "主队有利路径" 和 "客队风险点"
```

#### cautious_analyst：谨慎分析师

**角色设定**：看空主队的分析师，职责是找出所有不支持主队获胜的风险因素。

```
你是一位对 {主队} 持谨慎态度的分析师。
你的任务是：基于提供的数据，找出所有可能让 {主队} 失败的因素。
要求：
1. 每个风险点必须有数据支撑
2. 承认对你不利的数据，但指出隐患
3. 输出 "客队机会" 和 "主队风险点"
```

**关键设计**：两个分析师节点是**并行执行**的，各自独立分析，不受对方观点干扰。

#### synthesize_report：报告综合

接收两个分析师的输出，综合成一份完整的赛前分析报告：

- 总结双方核心论点
- 给出平衡后的胜率评估
- 列出关键胜负手
- 标注不确定性因素

```
你是一位资深体育主编。
你收到了两位分析师的报告（一位看好主队，一位看空主队）。
请综合两份报告，输出一份客观、平衡的赛前分析。

要求：
1. 同时呈现双方的核心论据
2. 给出你认为的胜率区间（必须说明判断依据）
3. 指出最关键的 2-3 个胜负手
4. 标注最大的不确定性因素
```

#### review_report：质量审查

独立的审查节点，检查报告是否满足质量标准：

- 是否有数据编造
- 逻辑是否自洽
- 是否过度自信或过度保守
- 是否遗漏重要维度

```
你是一位严格的质检编辑。
请审查以下分析报告，检查：
1. 是否有编造的数据
2. 逻辑推理是否成立
3. 结论是否过度自信
4. 是否遗漏了伤病、赛程等重要因素

输出：通过 / 不通过
如果不通过，说明具体问题和修改建议。
```

#### finalize：输出终稿

审查通过后，将报告格式化为最终输出结构。

### 条件边设计

```python
def review_router(state: AgentState):
    if state["review_passed"]:
        return "finalize"
    elif state["retry_count"] < 1:
        return "synthesize_report"
    else:
        return "finalize"  # 重试一次后仍不通过，强制输出
```

这个设计确保：

- 质量达标时直接输出
- 质量问题可回退修改（最多一次）
- 避免无限循环

## 四、工程实现要点

### 状态定义

```python
class AgentState(TypedDict):
    fixture: dict           # 比赛基础数据
    team_form: str          # 状态分析结果
    injuries: str           # 伤病分析结果
    matchup: str            # 对位分析结果
    bullish_analysis: str   # 乐观分析
    cautious_analysis: str  # 谨慎分析
    report: str             # 综合报告
    review_passed: bool     # 审查是否通过
    retry_count: int        # 重试次数
    final_output: str       # 最终输出
```

所有节点共享同一个 State 对象，每个节点读取需要的数据，写入自己的输出。

### 双数据源设计

项目支持 mock 数据和 ESPN 真实数据并存：

| 数据源 | 用途 | 特点 |
|--------|------|------|
| mock | 稳定演示 | 数据固定，不依赖外部接口 |
| ESPN | 真实分析 | 实时数据，网络依赖 |

前端通过标签区分：`[样例] 湖人 vs 勇士` vs `[真实] Brooklyn Nets vs New York Knicks`

### 两种运行模式

**rule-based 模式**：不需要 API Key，按预设规则生成结果。适合演示和测试。

**llm 模式**：需要配置 OpenAI 兼容接口，启用真正的 AI 分析。生产环境使用。

```bash
export OPENAI_API_KEY=你的key
export OPENAI_MODEL=gpt-4.1-mini
export OPENAI_BASE_URL=https://你的兼容网关/v1
```

### 前端展示

前端使用 Next.js + ReactMarkdown 渲染分析报告：

- 浅色主阅读区，深色正文文字
- 卡片式布局区分不同分析维度
- 主客队信息独立展示
- 风险标记用醒目颜色标注

## 五、Prompt 工程原则

### 一个节点一个角色

每个 Prompt 只赋予模型一个明确的角色，不混合多个角色。角色越单一，输出越聚焦。

### 结构化输入

向每个节点传递结构化的数据，而不是让模型自己判断哪些信息相关。

```python
# 好的做法：明确传递所需数据
prompt = f"""
【球队战绩】
{team_record}

【近五场结果】
{recent_games}

【伤病名单】
{injuries}
"""

# 不好的做法：让模型自己筛选
prompt = "这里有一堆数据，你自己看哪些有用。"
```

### 禁止编造

在 Prompt 中明确声明"禁止编造数据"，并在 review 节点检查是否违规。

### 审查节点独立

reviewer 不能是 writer 的"自己人"。独立的审查视角才能发现真正的问题。

## 六、测试与验证

### 单元测试

每个节点都可以独立测试：

```python
def test_bullish_analyst():
    state = create_mock_state()
    result = bullish_analyst(state)
    assert "主队有利路径" in result["bullish_analysis"]
    assert "客队风险点" in result["bullish_analysis"]
```

### 端到端测试

```bash
# CLI 入口测试
uv run python -m app.cli --match-id lakers_vs_warriors --style pro

# API 测试
GET /api/matches
POST /api/analyze
```

### 质量评估

长期运行的系统需要建立 LLM 输出质量的评估框架：

- 与赛后真实结果对比，统计胜率预测准确率
- 人工抽查报告质量，标记幻觉和逻辑错误
- A/B 测试不同 Prompt 版本的效果差异

## 七、可复用的模式总结

这个项目的多 Agent 工作流可以抽象为一个通用模式：

```
数据采集 -> 多维度分析 -> 多观点辩论 -> 综合报告 -> 质量审查 -> 终稿输出
```

这个模式适用于任何需要**多角度分析 + 质量把控**的场景：

- 投资研报生成
- 代码审查
- 产品需求评审
- 医疗诊断辅助

## 八、局限与改进方向

### 当前局限

- 真实数据源还不够"专业级"，缺少 advanced stats 和首发预测
- 前端仍是 MVP 级别，缺少球队 logo、战绩卡片等体育产品常见元素
- 样例数据和真实数据没有筛选器

### 改进方向

**P1：产品化**
- 增加球队 logo、战绩卡片、比赛状态标签
- 增加数据来源筛选（样例 / ESPN）
- 明确数据缺失时的降级策略

**P2：功能扩展**
- 接入更多真实数据源
- 增加"赛中实时分析"子图
- 增加"赛后复盘"子图
- 输出社媒短文案 / 视频口播稿

**P3：多联赛**
- NFL / 英超 / 欧冠扩展
- 用户收藏球队
- 个性化关注点模板

## 结语

LangGraph 的多 Agent 工作流不是"为了复杂而复杂"。当分析任务需要多角度观点、独立审查、可调试的中间过程时，状态机模型的价值就会显现。

NBA 赛前分析只是一个具体的应用场景。更关键的是掌握这套"分解问题 -> 分配角色 -> 编排流程 -> 质量把控"的方法论，它可以迁移到任何需要结构化 AI 协作的领域。
