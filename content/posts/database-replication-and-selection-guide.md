## 引言

数据库复制（Replication）是构建高可用、高性能、可扩展数据架构的基石。从最简单的热备份到复杂的分布式共识，复制技术的演进贯穿了数据库发展的整个历史。

本文从复制的分类出发，深入讲解 MySQL、PostgreSQL、Redis 的复制原理，分析主从延迟的根因与优化手段，最后给出数据库选型的系统方法论。

## 一、复制的分类体系

### 1.1 按同步方式分类

| 类型 | 机制 | 一致性 | 延迟 | 适用场景 |
|------|------|--------|------|---------|
| 同步复制 | 主库等待从库确认后才返回客户端 | 最强 | 最大 | 金融级核心交易 |
| 异步复制 | 主库写完本地日志即返回 | 最终一致 | 最小 | 互联网读多写少 |
| 半同步复制 | 至少等待一个从库确认 | 兼顾 | 中等 | 一般业务高可用 |

同步复制的致命问题是：从库故障或网络抖动会阻塞主库写入。生产环境更常用异步复制 + 监控告警的方案。

### 1.2 按日志格式分类

| 类型 | 代表 | 说明 |
|------|------|------|
| 物理复制 | PostgreSQL WAL Streaming | 复制磁盘页的物理变更，效率高 |
| 逻辑复制 | MySQL Binlog、MongoDB Oplog | 复制 SQL 语句或行变更，灵活性强 |

物理复制的优点是性能高、与存储引擎紧耦合；逻辑复制的优点是可跨版本、可筛选表/库级复制。

### 1.3 按拓扑结构分类

| 拓扑 | 说明 | 特点 |
|------|------|------|
| 主从（一主多从） | 最常用，读写分离 | 简单，从库可水平扩展读 |
| 级联复制 | 从库再带从库 | 减轻主库复制压力 |
| 多主复制 | 多个节点同时接受写 | 冲突复杂，适合多地域写入 |
| 环形/链式 | 较早的拓扑 | 故障排查复杂，现已少用 |

## 二、MySQL 主从复制原理

### 2.1 复制流程

```
Master
    ├── 事务提交 -> 写 Binlog（Dump Thread 推送）
    └── 等待 Slave ACK（半同步）

Slave
    ├── I/O Thread：接收 Binlog -> 写 Relay Log
    ├── SQL Thread：重放 Relay Log -> 应用到本地库
    └── 多线程并行复制（5.6+）：按库并行 / 按事务组并行
```

### 2.2 关键组件

| 组件 | 作用 |
|------|------|
| Binlog | 主库的逻辑日志，记录所有数据变更 |
| Relay Log | 从库本地缓存的 Binlog 副本 |
| Dump Thread | 主库上负责向从库推送 Binlog 的线程 |
| I/O Thread | 从库上接收 Binlog 并写入 Relay Log |
| SQL Thread | 从库上重放 Relay Log 的线程 |

### 2.3 GTID：全局事务标识

MySQL 5.6+ 引入 GTID（Global Transaction Identifier）：

```
GTID = source_id:transaction_id

例如：3E11FA47-71CA-11E1-9E33-C80AA9429562:23
```

GTID 的价值：

- 自动定位复制断点，无需手动指定文件和位置
- 简化主从切换（failover）
- 避免重复执行或遗漏事务

### 2.4 并行复制演进

| 版本 | 并行策略 | 说明 |
|------|---------|------|
| 5.6 | 按库并行 | 不同数据库的事务并行重放 |
| 5.7 | 按组提交并行 | 同一组提交内的事务无冲突，可并行 |
| 8.0 | Write Set 依赖并行 | 分析事务间的依赖关系，无依赖即可并行 |

并行复制的目标是解决"从库单线程重放跟不上主库多线程写入"的瓶颈。

### 2.5 主从延迟排查

**查看延迟**：

```sql
SHOW SLAVE STATUS\G
-- 关注：Seconds_Behind_Master、Slave_IO_Running、Slave_SQL_Running
```

**常见延迟原因**：

1. 主库写入吞吐量过高，从库单线程重放跟不上
2. 从库执行了大事务（如 DELETE 千万级数据），阻塞后续重放
3. 从库硬件配置低于主库，或从库同时承担大量读查询
4. 网络抖动或带宽不足，日志传输慢

**优化手段**：

1. 开启并行复制（slave_parallel_workers）
2. 大事务拆分为小事务
3. 从库独立部署，不混布其他重负载
4. 半同步复制降级为异步复制，避免主库被慢从库拖住

## 三、PostgreSQL 流复制

### 3.1 物理流复制

```
Primary
    ├── WAL Writer 持续产生 WAL 记录
    └── WAL Sender 进程向 Standby 推送 WAL

Standby
    └── WAL Receiver 接收 WAL -> 直接重放到数据页
```

PG 的流复制是**物理复制**，Standby 直接重放 WAL 到数据页，不需要像 MySQL 那样解析 SQL 再执行，效率更高。

### 3.2 同步复制级别

PostgreSQL 提供三级同步复制控制：

| 级别 | 含义 | 数据安全 | 延迟 |
|------|------|---------|------|
| remote_apply | 从库应用后才返回 | 最高 | 最大 |
| remote_write | 从库写到磁盘后返回 | 较高 | 中等 |
| on | 从库收到 WAL 即返回 | 较低 | 最小 |

### 3.3 逻辑复制

PostgreSQL 10+ 支持逻辑复制：

- 基于 Publication/Subscription 机制
- 可选表级复制，只复制需要的表
- 支持跨版本、跨平台复制
- 支持向异构目标（如 Kafka、Elasticsearch）复制

逻辑复制的适用场景：

- 数据仓库 ETL
- 跨数据中心的数据同步
- 数据库升级迁移

## 四、Redis 高可用架构

### 4.1 三种模式对比

| 模式 | 数据分片 | 自动故障转移 | 适用场景 |
|------|---------|------------|---------|
| 主从复制 | 否 | 否 | 简单读写分离 |
| Sentinel | 否 | 是 | 高可用 |
| Cluster | 是 | 是 | 大规模分布式缓存 |

### 4.2 主从复制

```
Slave 向 Master 发送 SYNC/PSYNC
Master fork 子进程生成 RDB 快照
Master 持续发送增量命令
Slave 加载 RDB 后执行增量命令
```

```bash
INFO replication
# 关注：master_repl_offset 和 slave_repl_offset 差值
```

### 4.3 Sentinel

Sentinel 是 Redis 的高可用方案：

- 监控主从节点的健康状态
- 自动故障转移：主库故障时，选举从库晋升为主库
- 通知客户端新的主库地址

最小部署：3 个 Sentinel 节点 + 1 主 + 2 从。

### 4.4 Cluster

Redis Cluster 是分布式方案：

- 数据分片：16384 个 Slot，每个 Slot 对应一个主从结构
- 客户端直连数据节点，不需要代理层
- 支持在线扩容和缩容

**为什么不支持跨 Slot 事务**：数据分布在不同节点上，没有分布式事务协议支持。需要在同一个 Slot 内的 key 才能使用事务。

## 五、数据库选型方法论

### 5.1 选型决策树

```
是否需要强事务 + 复杂查询 + 多表 JOIN？
    -> 是 -> 关系型（MySQL / PostgreSQL / Oracle）
    -> 否 -> 继续判断

数据结构是否高度灵活，需要嵌套文档？
    -> 是 -> MongoDB
    -> 否 -> 继续判断

是否是缓存、排行榜、会话等实时场景？
    -> 是 -> Redis
    -> 否 -> 继续判断

是否是全文搜索或日志分析？
    -> 是 -> Elasticsearch
    -> 否 -> 继续判断

数据规模是否超出单机存储上限？
    -> 是 -> 分布式方案（TiDB / HBase / ClickHouse）
    -> 否 -> 关系型单机即可

是否是 OLAP 海量聚合分析？
    -> 是 -> ClickHouse / Doris / StarRocks
    -> 否 -> 回到起点重新审视需求
```

### 5.2 选型对比表

| 场景特征 | 推荐选型 |
|----------|---------|
| 强事务、强一致性、复杂查询、多表 JOIN | 关系型（MySQL / PostgreSQL / Oracle） |
| 数据结构多变、高写入、嵌套文档、快速迭代 | 文档型 NoSQL（MongoDB） |
| 超大规模日志、时序数据、海量键值查询 | 宽列存储（HBase / Cassandra） |
| 缓存、排行榜、会话、实时计数 | 内存型 KV（Redis） |
| 全文搜索、日志分析、APM 链路追踪 | 搜索引擎（Elasticsearch） |
| HTAP、分布式强一致 SQL | NewSQL（TiDB / CockroachDB） |
| OLAP、海量数据聚合分析 | 列式数据库（ClickHouse / Doris / StarRocks） |

### 5.3 选型时的常见误区

| 误区 | 正确做法 |
|------|---------|
| "分布式一定比单机好" | 分布式带来网络延迟和复杂度，数据量不大时单机更优 |
| "一种数据库打天下" | 不同场景用不同数据库，微服务架构下多库共存是常态 |
| "追求最新技术" | 生产环境优先选择成熟稳定的方案 |
| "忽视运维成本" | 分布式数据库的运维复杂度远高于单机 |

## 六、复制架构的最佳实践

### 6.1 读写分离的正确姿势

| 层面 | 做法 |
|------|------|
| 应用层 | 通过数据库中间件（如 MyCat、ShardingSphere）或 ORM 配置读写分离 |
| 连接池 | 配置主库连接池和从库连接池，按 SQL 类型路由 |
| 延迟处理 | 写入后立刻读的场景，强制走主库，避免复制延迟导致数据不一致 |

### 6.2 故障切换 checklist

- 确认主库确实不可恢复
- 选择数据最新的从库（对比复制位点）
- 将从库提升为主库，重置复制关系
- 更新应用的数据库连接配置
- 修复旧主库后，作为新从库接入

### 6.3 监控指标

| 指标 | 说明 |
|------|------|
| Seconds_Behind_Master | MySQL 从库延迟秒数 |
| replay_lag / flush_lag | PostgreSQL 复制延迟 |
| master_repl_offset - slave_repl_offset | Redis 复制偏移量差值 |
| 复制线程状态 | IO/SQL 线程是否 Running |

## 结语

数据库复制不是简单的"数据拷贝"，而是涉及一致性模型、故障恢复、性能权衡的复杂工程问题。理解复制的底层原理——日志格式、同步方式、拓扑结构、并行策略——才能在面对延迟、故障、扩展需求时做出正确的架构决策。

选型时没有银弹，关键是**把业务需求映射到数据库的能力模型上**。强事务走关系型，海量写走 LSM-Tree，实时缓存走内存，全文搜索走倒排索引，OLAP 走列式存储。能力边界清晰，组合使用得当，才能构建稳健的数据架构。
